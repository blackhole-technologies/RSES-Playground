## Decisions made this session

Recorded so they're not re-litigated:

- **feature_flags schema deviates from SPEC §4.7 in four columns** — adds `toggleable`, `tags` (jsonb), `owner`, `sunset_date`. Without these, vendor/02's FeatureFlag couldn't round-trip without lossy reconstruction (in particular, "core" flags marked `toggleable: false` would silently become UI-toggleable). Same "reality > spec" precedent as Phase 1's `bootstrap_token_hash` deviation.
- **`feature_rollout_history.user_id` uses `ON DELETE SET NULL`** (not CASCADE). Same audit-preservation precedent as `invite_codes.consumed_by`.
- **Migration runner directory list** now includes `engines/*/migrations` alongside `core/migrations` and `modules/*/migrations`. Updated `bin/migrate.ts` and `bin/reset-db.ts`. Filename ordering still applies globally — `0001_auth_init` < `0001_core_init` < `0001_feature_flags_init` lexicographically, which keeps `users(id)` ahead of `feature_rollout_history`'s FK.
- **Storage adapter round-trip caveats** documented in `engines/feature-flags/storage.ts` header. Vendor-only fields not persisted: `abTestConfig`, `documentationUrl`, `issueKey`, `introducedInVersion`, `lastModifiedBy`, `dependents` (computed), `changeHistory` (lives in `feature_rollout_history`). NULL `description` maps to `""` so vendor's Zod doesn't reject reads.
- **`createAdminRouter` constructs the flag storage internally.** Couples admin to feature-flags at the engine level, but it's a natural coupling — admin's whole purpose on the flags surface is exposing flag state to operators. Keeps the factory signature unchanged so existing callers (bootstrap.ts, settings tests) needed no edits.
- **`SystemSettingsView` is a deliberate projection.** `bootstrap_token_hash` and `bootstrap_token_consumed_at` are NOT in the `/api/admin/settings` response shape so a stolen admin session cannot read them. Admin tooling that needs bootstrap state queries the DB directly.
- **`requireAdmin`** lives next to `requireAuth` in `modules/auth/middleware.ts`. Returns 401 for missing user, 403 for non-admin (defense in depth — if a route were misconfigured with `[requireAdmin]` alone, it still does the right thing). Always pair `[requireAuth, requireAdmin]` for proper 401-then-403 ordering.
- **Anti-enumeration on /register's invite mode:** every invite-code failure (missing, unknown, expired, already consumed) returns the same `400 invalid_invite_code`. An attacker cannot probe to distinguish "valid format but wrong" from "invalid format". Mirrors /setup's 404 treatment.
- **Per-IP rate limit reuses the existing limiter shape.** `createIpRateLimiter` wraps `createLoginRateLimiter` with SPEC §5.2's defaults (3 attempts / 1 hour). The `recordFailure` method is misnamed for IP usage (every attempt consumes a slot, not just failures), but a wider rename was scope creep. `recordHit` would be clearer; future caller sites can adopt that alias if it grows.
- **No public /register without an `ipRateLimiter` in production.** Bootstrap creates one and passes it via `AuthRoutesConfig.ipRateLimiter`. The config field is optional so tests that omit it fall through to unlimited register, but the production path always wires it.
- **Two layers of Zod on /register.** Loose at the route (just shape) and strict in the service (`RegisterUserInputSchema` enforces the regex / min-length). Inner wins; the outer just gives 400 a stable shape if the body is wildly malformed.

## What's next

**Phase 3 — Content scaffolding + Fields API + taxonomy.** Per `cms-spec/IMPLEMENTATION.md` §3.1 (Phase 3). Goal: content table + vocabularies/terms exist; user posts can be created/listed; faceted queries work. Per the spec, "do these together because they interlock tightly."

Five sub-pieces from the spec:

1. **`engines/fields`** — Fields API per DESIGN.md §2.5. Just a registry. Unit-testable (no DB).
2. **`engines/render`** — Render engine per DESIGN.md §2.6. Dispatch by `content_type_id + target`. Unit-testable.
3. **`engines/taxonomy`** — wraps `vendor/10-taxonomy-algebra` + `vendor/11-taxonomy-query-engine`. Loads vocabularies + terms + content_terms + content into `TaxonomyQueryEngine` at boot. Subscribes to `terms.changed` for invalidation. Interface from DESIGN.md §3.7.
4. **`modules/taxonomy`** — tables `vocabularies`, `terms`, `content_terms`. Routes: user CRUD + admin system-vocab CRUD. Seed one starter system vocabulary `topics` with a few terms during install.
5. **`modules/content`** — tables `content_items`, `user_item_state`, `content_item_dedup_hash`. Register content type `post` with Fields API. Register renderer for `post` with Render engine. Full `/api/content/*` routes. Public `GET /p/:username/:slug`. Job handlers `content.retention-purge`, `content.soft-delete-purge`.

Install order: `auth` → `taxonomy` (seeds vocabularies) → `content` (references them).

Completion criteria from spec test list:
- `POST /api/content { title, body_markdown }` → 201 with rendered html
- `GET /api/content` → returns the post
- `PATCH /api/content/:id` with updated markdown → HTML updates
- `POST /api/vocabularies` + `POST /api/terms` + `POST /api/content/:id/tags` chain works
- `GET /api/content?terms=<id>` → returns tagged post
- `DELETE /api/content/:id` soft-deletes → `GET` 404s; manual purge job hard-deletes

This is bigger than Phase 2. Plan ~10–14 commits across the five sub-pieces.

## Gotchas

- **Phase 3 introduces user-scoped content tables for the first time.** Every content_items / user_item_state row belongs to a user. Register them with `tenantScope.register(table, table.userId)` in bootstrap. The 20-tuple isolation property test from Phase 1 only covers `sessions`; consider adding equivalent assertions for content tables once they're live.
- **Vendor/10 and vendor/11 expect to be initialized from a full snapshot.** The taxonomy engine's boot path loads everything from the DB into the in-memory `TaxonomyQueryEngine`. If the DB grows large this becomes slow; for MVP it's fine. Keep an eye on it.
- **Fields API is dependency-free.** No DB, no event bus. Just a typed registry. Should land first as a self-contained engine.
- **Render engine dispatches by `(content_type_id, target)`.** Targets in scope for Phase 3: `'public-html'` (the user-facing page render) and `'rss-atom'` (for outgoing feeds, even though Phase 5 is where outgoing feeds wire up). Register both for `post`.
- **Slug generation is on the user side** — content uses `(username, slug)` as the public URL key. SPEC mandates per-user slug uniqueness. New functional unique index probably needed (`UNIQUE (user_id, slug)` or partial). Verify against SPEC §4.3.
- **Soft-delete purge is on a daily cron.** `engines/queue` already exists in the codebase (vendor/08 wrap) but isn't wired into bootstrap yet. Phase 3's content module needs the queue running. Probably wire that in before content.
- **Public `GET /p/:username/:slug`** is unauthenticated — the auth middleware is permissive so it'll fall through with `req.user` undefined; the route just resolves to the user record by username and serves the rendered HTML. No `requireAuth` on this one.
- **`pnpm test:integration` is at ~30s and growing.** Each integration file does its own `DROP SCHEMA + migrate` in beforeAll. If Phase 3 adds 6+ integration files, consider extracting setup to a vitest `globalSetup` that runs once per session.
