## Decisions made this session

Recorded so they're not re-litigated:

- **Bootstrap token storage:** the DB stores `sha256(token)` only, not raw. The raw token is logged exactly once in the `FIRST BOOT: visit ...` line and never appears anywhere else. The previous handoff's migration already had `bootstrap_token_hash` + `bootstrap_token_consumed_at` rather than SPEC Â§4.1's literal `bootstrap_token`; service.ts adopts that schema.
- **Username case-folding:** option 1 â€” functional unique indexes `lower(username)` + `lower(email)`. The email index is partial (`WHERE email IS NOT NULL`) so multiple users can omit email without colliding. No `citext` extension needed. Queries MUST use `WHERE lower(username) = lower($1)`; a raw `username = $1` silently bypasses case-folding.
- **`invite_codes.consumed_by` on consumer user delete:** `ON DELETE SET NULL`. Preserves the audit row so a consumed code cannot be reissued even after the consumer is deleted. SPEC Â§4.1 was ambiguous on this FK; we chose SET NULL.
- **Rate limiter shape:** in-memory, per-username, single-process. Failure counters reset on restart and don't propagate across instances. The `LoginRateLimiter` interface accommodates a Redis/DB backend swap when horizontal scaling demands it; for now Phase 1 ships with the in-process Map. Failures during an active lockout do NOT extend the lockout window â€” that would let an attacker keep an account locked indefinitely.
- **Cookie strategy:** `__Host-session` + `Secure` in production; `session` (no Secure) in dev/test. SameSite=Strict + HttpOnly unconditionally. Centralized in `bootstrap.ts` so routes/setup-routes don't drift.
- **Public `/api/auth/register`:** deferred to Phase 2 because `system_settings.registration_mode` defaults to `disabled` and the mode toggle UI is Phase 2 territory. Phase 1's only user-creation path is `/setup` for the bootstrap admin.
- **`USER_COLUMNS` is a function, not a constant.** `userColumns(alias?)` accepts an optional table-alias prefix because `loadSessionUser`'s JOIN exposed the ambiguity of bare `id` (sessions.id vs users.id). Single-table queries pass no alias; joined queries pass `"u"`.
- **Anti-timing-attack in verifyCredentials is documented but not implemented.** When the username doesn't exist, no scrypt hash runs â€” observable as a faster response. Per-username rate limiting (5 failures / 15 min) renders the attack impractical in practice; a precomputed dummy hash for missing users would close the leak fully.
- **Test infrastructure:**
  - `docker-compose.yml` pins `postgres:16-alpine`, seeds `loom` + `loom_test` databases via `scripts/init-test-db.sql` on volume init.
  - `--no-file-parallelism` on `test:integration` because three integration files do `DROP SCHEMA + migrate` in beforeAll; parallel workers race against the shared `loom_test` DB.
  - `tsx --env-file=.env` wired into `dev` / `db:migrate` / `db:reset` so `cp .env.example .env` is enough to run cold â€” no manual exports or direnv.
  - `.gitignore` got a `!.env.example` exception. Without it, the `.env.*` rule was silently shadowing `loom/.env.example` and edits never reached version control.
- **Migration runner ordering:** sorts globally by filename, not by directory. `0001_auth_init.sql` actually runs BEFORE `0001_core_init.sql` because `'a' < 'c'`. Acceptable today (auth has no FK dependencies on core tables); future cross-module dependencies must use `0002_*` or later to force the correct order.

## Gotchas

- **`requireAdmin` cleanup belongs in `modules/auth/middleware.ts`**, not `modules/admin/`. The check (`req.user.isAdmin`) belongs to the auth domain even though the only consumers will be admin routes; otherwise non-auth modules would have to import from `modules/admin/`.
- **Invite-code consume is the same `FOR UPDATE` pattern as `consumeBootstrapToken`.** Don't try to be clever with optimistic concurrency control â€” the row lock is the correct tool. Two concurrent registrations with the same code must serialize, with one winning the consume and the other observing `consumed_by IS NOT NULL`.
- **IP-based rate limit is shaped differently from per-username.** SPEC Â§5.2's "3 per IP per hour" for `open`-mode register is a different keying. Easiest: generalize `LoginRateLimiter` to take an arbitrary string key and instantiate two limiters (`loginByUsername`, `registerByIp`). Bootstrap creates both.
- **Feature-flags engine wraps `vendor/02-feature-flags`.** Look at `vendor/02-feature-flags/src/storage.ts` â€” it's an interface; the engine provides a Drizzle-backed implementation. The flag-evaluation logic itself stays in vendor/02; only the storage adapter is new.
- **Admin module's `manifest.ts` is mentioned in IMPLEMENTATION.md** but module-loader still doesn't exist (Phase 1 punted it; bootstrap directly mounts auth). Defer manifest.ts to whenever module-loader lands.
- **`pnpm test:integration` is now 15s and growing.** Each integration file does its own `DROP SCHEMA + migrate` in beforeAll. If Phase 2 adds 3+ more integration files, consider extracting setup to a vitest `globalSetup` that runs once per session.
- **Bootstrap log line for the FIRST BOOT token uses `log.warn` (not `log.info`).** The handoff specifies "log to stdout exactly once" but doesn't dictate level; warn was chosen to make the line stand out in default-INFO log streams. Don't change to info or it'll get drowned out.
