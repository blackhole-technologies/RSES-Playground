## Gotchas

- **`/setup` wrong-token returns 404.** Easy to accidentally change to 401 during a refactor. Test it explicitly.
- **Bootstrap token concurrency:** two simultaneous `POST /setup` with the same valid token must fail cleanly — one wins, the other gets 404 (not 500). Use `SELECT … FOR UPDATE` inside the `consumeBootstrapToken` transaction.
- **Admin password reset is CLI-only** (SPEC §5.3). Do not add an HTTP reset route — there's no SMTP dependency and no phishing surface by design. Reset via a future `bin/reset-admin.ts`.
- **Session cookie flags:** `__Host-` prefix in production, `SameSite=Strict`, `httpOnly`, `Secure`. Test cookie flags in both `NODE_ENV=production` and `test`.
- **Tenant-scope integration test:** needs a live DB. Put at `loom/engines/tenant-scope/scoped.integration.test.ts` (or under `loom/tests/integration/tenant-scope.test.ts`) with `describe.skipIf(!process.env.TEST_DATABASE_URL)`.
- **`app.current_user_id` is a string session var.** RLS policies read via `current_setting('app.current_user_id', true)`. No type coercion — store userId as TEXT or cast at read time.
- **Dev-guard does not intercept `db.transaction(fn)`-returned `tx`.** That's the escape hatch. `scoped()` and `withDbUserScope` both go through transactions, so they naturally bypass the guard. A bug where someone uses raw top-level `db.select().from(users)` in dev throws loudly. In prod (no guard), RLS handles it.
- **Drizzle version in loom:** `0.36.0`, not the parent's `0.39.3`. If any patterns you remember from parent don't work, check Drizzle's 0.36 docs.
