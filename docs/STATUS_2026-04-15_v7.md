## Phase D readiness audit (the new work in v7)

### What was audited

Every call site in `server/` that touches `site_feature_overrides` or `feature_rollout_history` outside of the `scoped()` or `withDbSiteScope()` helpers. Grep targets:

- `siteFeatureOverrides` as a Drizzle table import or query argument.
- `featureRolloutHistory` as a Drizzle table import or query argument.

The audit found **100% of the raw-`db` accesses clustered in a single file**: `server/services/feature-flags/pg-storage.ts`. Nothing else in `server/` queries these two tables directly. The two classes in that file — `PgSiteOverrideStorage` and `PgRolloutHistoryStorage` — implement the `ISiteOverrideStorage` and `IRolloutHistoryStorage` interfaces consumed by `FeatureFlagService` (`server/services/feature-flags/index.ts`) and the flag `Evaluator` (`server/services/feature-flags/evaluator.ts:454` — the flag-evaluation hot path).

**Bucket 3 — policy/design mismatch (resolved via migration 0005):**

| Site | Line | Issue |
|---|---|---|
| `PgRolloutHistoryStorage.record(event)` (when `event.siteId == null`) | 758–771 | Writes a global-scope audit row (`siteId = null`). ### How Bucket 3 was resolved

Two-line migration 0005 (`DROP POLICY` + `CREATE POLICY`) replacing the strict INSERT check on `feature_rollout_history` with a disjunctive form that matches the SELECT policy:

```sql
CREATE POLICY "site_isolation_insert" ON "feature_rollout_history"
  FOR INSERT
  WITH CHECK (
    site_id IS NULL
    OR site_id = current_setting('app.current_site_id', true)
  );
```

This exactly mirrors the 0003 SELECT policy. Net effect on the threat surface (full analysis in the migration file):

- **Accepted:** A tenant-bound request can now insert NULL-site_id rows. Mitigation: Layer 1 (route auth + RBAC markers) is the primary gate on who reaches `record()` at all; non-admins cannot call `createFlag` / `addDependency` / `removeDependency`.
- **Preserved:** A tenant-bound request still cannot forge a row with a *different* non-null site_id (the second disjunct enforces equality).
- **Preserved:** SELECT semantics unchanged (global rows remain readable by all tenants; per-site rows remain isolated).
- **Preserved:** UPDATE/DELETE policies unchanged (tenants still cannot modify or delete global rows).

Two alternatives were considered and rejected:

1. **Option 2 — superuser connection for global writes.** Cleaner Layer-3 posture but requires a new DB client, env var (`SUPERUSER_DATABASE_URL`), connection pool, and doubled test surface. Rejected because the security benefit is contingent on an attacker reaching `record()` without passing Layer 1, which would itself be a separate bug this migration cannot fix.
2. **Option 3 — sentinel siteId.** Rejected because it changes the semantic contract of `feature_rollout_history.site_id` (NULL = global), requires a data migration for existing rows, and leaks into Drizzle schema + shared types + React admin UI.

### Test coverage

The RLS integration test (`tests/rls/rls-isolation.test.ts`) was updated in lockstep with migration 0005:

- Assertion 10 changed from "INSERT with NULL site_id is rejected" to "INSERT with NULL site_id is allowed (global audit write)".
