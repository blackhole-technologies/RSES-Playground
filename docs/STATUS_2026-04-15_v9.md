### Storage-layer conversion

| Call site | Treatment |
|---|---|
| `createKey(siteId, ...)` | Converted to `withDbSiteScope(siteId, tx => insert.returning())` because scoped() does not surface the returned row |

| `revokeKey(siteId, keyId)` | **Signature changed** — now takes `siteId` as first arg. Uses `withDbSiteScope` for the update-returning rowCount. Old signature was `revokeKey(keyId)`; the deprecated wrapper in `sdk-api.ts:revokeApiKey` updated in lockstep |
| `listKeys(siteId, includeRevoked)` | Converted to `withDbSiteScope` because scoped().select does not support `orderBy(apiKeys.createdAt)` |
| `getKey(siteId, keyId)` | **Signature changed** — now takes `siteId` as first arg. Uses `scoped(siteId).selectOne(apiKeys, { id: keyId })` |
| `updateLastUsed(keyId, siteId)` | **Signature changed** — now takes `siteId` as second arg. `validateKey` passes the row's siteId after the cross-tenant hash lookup resolves it. Uses `scoped(siteId).update(...)` |
| `cleanupExpiredKeys(retentionDays)` | Kept as raw `db.delete()` with a comment marking it **INTENTIONAL CROSS-TENANT** (background GC across every site). Documented as not currently wired to any scheduler |

Live runtime call path today: `apiKeyAuth` middleware → `ApiKeyService.validateKey(key)` → on success, async `ApiKeyService.updateLastUsed(keyId, siteId)`. Everything else (`createKey` / `revokeKey` / `listKeys` / `getKey`) is reachable only via the three `@deprecated` wrappers at the bottom of `sdk-api.ts` which are not imported anywhere. The guard still enforces the conversion on all paths — if a future PR wires `getKey` or `revokeKey` into a live admin route, the scoped path is already in place.

### Migration 0006 (STAGED, NOT APPLIED)

`migrations/0006_add_api_keys_rls_policies.sql` — 4 policies with strict site_id equality on SELECT / INSERT / UPDATE / DELETE. `ENABLE ROW LEVEL SECURITY` only (no FORCE), same posture as 0003. Multi-paragraph apply-procedure banner with the **validateKey problem** section, which documents three deployment-time options for keeping the SDK auth flow working once policies land:

1. **Superuser connection for validateKey** — simplest, widens superuser blast radius.
2. **Dedicated `app.api_key_lookup` session variable** (recommended) — add a 5th bypass policy in a later migration 0007; wrap validateKey's SELECT in a small helper that sets the flag. Keeps validateKey on the app's main DB client and survives a future FORCE flip.
3. **Encode site in the key itself** — rejected during M1.8a because it changes the key format wire protocol.

0006 ships only the four strict policies. The user picks option (1), (2), or (3) at apply time. 0006 is **NOT applied by default** — it is reversible (DROP POLICY + DISABLE ROW LEVEL SECURITY restores the pre-migration state) and does not cascade onto any existing data.

## The validateKey cross-tenant problem (unresolved by design)

Discovered and documented during M1.8a. The SDK auth flow arrives at the server with an opaque API key and NO site context. The owning site is learned only AFTER the key is looked up by its hash. Under the strict equality SELECT policy in migration 0006, an unbound session variable returns the empty string and the comparison fails — every valid API key lookup returns zero rows, and the SDK sees 401 Unauthorized.

v9 chose NOT to resolve this in code because the resolution is a deployment-architecture choice, not a pure code fix. The three options are enumerated in both:

- `migrations/0006_add_api_keys_rls_policies.sql` (full rationale with code sketches for each option).
- `server/services/api-keys/api-key-service.ts:validateKey` (operational note pointing at the migration).
- `tests/rls/rls-isolation.test.ts` (assertion 3 in the api_keys block, with a comment explaining it documents the runtime break).

The user must decide option (1) / (2) / (3) before applying 0006 to production. Staging apply of 0006 without resolving the validateKey problem is safe — staging probably uses a superuser DB role and will behave identically to dev (ENABLE policies bypassed for superuser), surfacing the issue only when a non-superuser role is used.

## Apply sequencing (updated)

v9 adds a new migration to the pending stack. The full current pending order is:

1. **0005 first** (relax feature_rollout_history INSERT) — unchanged from v7.
2. **0006 second** (api_keys strict equality) — **NEW in v9**. Independent of 0004 and 0005; can ship on its own.
3. **0004 third** (FORCE on site_feature_overrides + feature_rollout_history) — unchanged from v6. Depends on 0005 being applied first.
4. **[Future] 0007** — if the user picks option (2) from the validateKey problem, a follow-up migration adds the `api_key_lookup` bypass policy.
5. **[Future] 0008** — if the user wants FORCE on api_keys as well, a follow-up migration mirroring 0004 but on api_keys. Depends on 0007 or a superuser connection being in place first.

0005, 0006, 0004 are mutually independent migrations in terms of the *tables* they touch, but 0004 depends on 0005 for correctness (the `feature_rollout_history` admin writes break under 0004 without 0005). 0006 is independent of both.

### Recommended apply order

Simplest path: **0005 → 0006 → 0004**. This lands both relax migrations before the FORCE flip, keeping the apply sequence reversible at every step.
