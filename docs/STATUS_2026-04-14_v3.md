### TypeScript error cleanup (691 → 175, -75%)

The v1 audit reported 18 TypeScript errors. They were all in two files
(`useDesignSystem.ts`, `use-accessibility.ts`) that contained JSX in `.ts`
files — syntax errors so severe that TypeScript bailed out of the parse and
silently skipped type-checking everything those files transitively
depended on.

In the 2026-04-14 evening pass, those two files were renamed `.tsx`
(safe — every importer used extensionless paths). The rename revealed the
true baseline of **691 type errors** that had been hidden behind the
syntax errors. Cascade root-cause analysis identified seven patterns that
drove ~80% of the count:

| # | Pattern | Files affected | Errors fixed | Fix |
|---|---|---|---|---|
| 1 | Enum types under `import type` but used as runtime values | workflow-engine, reactive-streams, saga, integration-connectors, ai-module-system | ~120 | Move enums out of `import type` block into a value import |
| 2 | Duplicate trailing `export { … }` blocks where symbols were already inline-exported | 7 sync/* files, infrastructure-adapters, 2 cms type files | ~140 | Delete the trailing block, replace with a code comment |
| 3 | Express `req.params.X` typed as `string \| string[]` due to buggy `@types/express-serve-static-core` `ParamsDictionary` declaration | cms/routes, 11 routes/* files, feature-flags/routes, feature-flags/site-routes | ~67 | Wrap each access in `String(…)` via per-file `replace_all` Edits |
| 4 | `shared/messaging/index.ts` ambiguously re-exported the same type names from `./types`, `./schemas`, and `./database-schema` | shared/messaging/index.ts | 24 | Replace `export *` from schemas/database-schema with explicit re-exports of just the constants |
| 5 | `cqrs-es/index.ts` initializer functions used local references that were re-exported but never imported into local scope | cqrs-es/index.ts | 19 | Add a local `import { getEventStore, … } from "./*"` block |
| 6 | `BaseSubscription.onRequest`/`onCancel` were `protected` but the publisher classes assigned to them externally | reactive-streams.ts | 8 | Change to `public` with a doc comment explaining the publisher-collaboration pattern |
| 7 | `WSMessageType` union missing 9 collaboration message types used by `use-collaboration.ts` | client/src/hooks/use-collaboration.ts | 9 | Extend the union in `server/ws/types.ts` |

Plus 4 isolated fixes:
- `kernel/events.ts`: `HandlerRegistration<T>` → `<unknown>` covariance — cast through `unknown`.
- `lib/tenant-scoped.ts`: 3 Drizzle builder type assertions tightened (cast through `unknown`).
- `cross-site-orchestration.ts`: widen `CrossSiteMessage.type` from a narrow string union to `string` to accommodate the runtime enum.
- `client/src/design-system/index.ts`: add local `import type { ResponsiveToken, ContextualToken, ContentContext }` so functions defined later can reference them.

**Result: 691 → 175 errors. Zero errors in any file added or modified by M0/M1/v3.**

### M1.5.4 — First route file migrated to fail-closed RBAC markers

- `server/routes/admin-audit.ts` rewritten to use `protect("audit_logs:read", …)` for every handler. Removes the `requireAuth` + `requireAdmin` + `AuditPermissions.read` triple-stack with a single marker per handler. Behavior is unchanged (still admin-only via the audit-logs permission, audit logging still fires on deny, fail-closed on RBAC service error).
- The migration is the **template** for the other route files. Pattern is documented in the file header.

### M1.5.5 — Marker coverage lint test

- `tests/security/rbac-marker-coverage.test.ts` (new): scans every file in `MIGRATED_FILES` for `router.METHOD(...)` calls and asserts each handler is wrapped in one of the marker functions. Uses a small balanced-bracket parser to walk multi-line handler arguments.
- 4 tests, all passing. Failures point to specific line numbers with a remediation message.
- New file migrations get a single line added to `MIGRATED_FILES`. The lint then enforces coverage on every PR.

### M1.6 — Security tests as CI gate

- `.github/workflows/ci.yml` updated. Three jobs:
  1. **`security-tests`** — runs `npm run test:security` with no DB. ~15 seconds. Hard gate. Required check.
  2. **`typecheck`** — runs `npm run check`. `continue-on-error: true` until the 175 residual errors reach zero, then becomes a hard gate. Surfaces the count as a soft warning today.
  3. **`build-and-test`** — depends on `security-tests` so a security failure shortcircuits the build and saves CI minutes.
- A failing security test now blocks merge to `main` (after branch protection picks up the new required check).
