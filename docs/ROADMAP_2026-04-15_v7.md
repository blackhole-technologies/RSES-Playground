### Exit criteria

- ✅ Dev-mode run throws a loud error on any raw `db.*` access to a tagged table.
- ✅ Production and test paths pay zero Proxy overhead (`enabled: false` returns the raw db by identity).
- ✅ Zero TypeScript regressions. Drizzle's deep type surface at the `db` export is preserved.
- ✅ New unit-test file stays in the fast suite (no DB dependency).
- ✅ Existing `scoped()` / `withDbSiteScope()` escape hatches continue to work — they use a `tx` from inside `db.transaction()`, which is not the same object as the wrapped `db`.
- ✅ Static lint `tenant-scoped-imports.test.ts` still passes — the guard is a complement, not a replacement.

### What the guard does NOT do

- Does NOT block anything in `NODE_ENV=test` or `NODE_ENV=production`. Test-time blocking would require rewriting every fixture; prod-time blocking duplicates the Postgres RLS defense (and risks false positives bricking reads).
- Does NOT intercept `db.query.tableName.findMany(...)` (Drizzle's relational-query API). That API is not used anywhere in `server/` at time of v7 — grep confirmed zero matches. If a future feature starts using it, add interception at `get(target, "query")` and recursively wrap the returned object.
- Does NOT register any tables itself. Registration is still via `tenant-scoped-registry.ts`, same as before. The guard only *reads* the registry.

## Milestone 1.8 — Expand tenant-scoped table registry (**REVISED: 1–3 days, not 0.5 day**)

### Why the scope estimate changed

v6 assumed a small number of unregistered tables. The v7 session ran a full grep of `shared/` for `site_id` columns and found **at least ~15** tables across five schema files that are not yet registered:

- `shared/social-media-schema.ts` — 3 tables (`social_accounts`, `social_posts`, `campaigns`), `site_id NOT NULL`.
- `shared/rbac-schema.ts` — user-role scopes, user-permission scopes, audit log. Mostly NULLABLE `site_id` because some assignments are global.
- `shared/api-keys-schema.ts` — 1 table (`api_keys`), `site_id NOT NULL`.
- `shared/multisite-schema.ts` — 5+ tables (`domains`, `role_assignments`, analytics, one table with both source and target site columns for site replication, plus one NULLABLE).
- `shared/schema.ts` — 2 tables (one NOT NULL, one NULLABLE).
- `shared/admin/schema.ts` — `site_metrics` (NOT NULL) and one NULLABLE table beyond the ones already registered.

Each of these needs the same three-layer treatment: registered in `tenant-scoped-registry.ts`, covered by a policies migration (new file `0006_*` — 0003 is immutable), and tested in `rls-isolation.test.ts`. Plus an audit of the existing `server/` code that queries each for direct-bypass call sites (same pattern as the v7 `pg-storage.ts` audit).

### Scope breakdown

| Sub-task | Time |
|---|---|
| 1.8.1 Grep `server/` for raw `db.*` access against each candidate table — one pass per table file | 0.5 day |
| 1.8.2 Classify each call site (per-site / cross-tenant / policy mismatch) — same buckets as v7 | (included in 1.8.1) |
| 1.8.3 Write migration `0006_expand_tenant_rls_policies.sql` covering NOT-NULL-site_id tables with strict equality and NULLABLE-site_id tables with the 0005 disjunctive INSERT form | 0.5 day |
| 1.8.4 Register each table in `tenant-scoped-registry.ts` | 0.1 day |
| 1.8.5 Extend `MULTI_TENANT_TABLE_NAMES` in `tenant-scoped-imports.test.ts` for the static lint | 0.1 day |
| 1.8.6 Convert per-site bypasses in `server/` (same pattern as v7's `pg-storage.ts` conversion) | 0.5–1 day depending on call-site density |
| 1.8.7 Mark cross-tenant admin paths with comments | (included in 1.8.6) |
| 1.8.8 Extend `tests/rls/rls-isolation.test.ts` with assertions for each new table (select / insert / update / delete + NULL handling where relevant) | 0.5 day |
| **Total** | **~1–3 days** |

### Dependencies

- **Should land after Phase D applies** to staging/production. Otherwise 1.8.3 has no deployed enforcement — the new policies would sit in a migration file with nothing under FORCE to back them.
- **Depends on M1.3 being in place** (which v7 shipped) so the guard catches any bypass missed during 1.8.6.
- **Depends on migration 0005 being applied** so the disjunctive-insert pattern can be reused in 0006 for NULLABLE-site_id tables without repeating the Phase B mistake.

### Sub-milestone splitting

M1.8 is large enough that it can ship as a chain of smaller PRs, one schema file at a time:

- **M1.8a**: `shared/api-keys-schema.ts` (1 table, smallest, good to validate the playbook).
- **M1.8b**: `shared/social-media-schema.ts` (3 tables, all NOT NULL site_id — straightforward).
- **M1.8c**: `shared/multisite-schema.ts` (5+ tables, one of which has source/target site columns — novel policy shape, handle carefully).
- **M1.8d**: `shared/rbac-schema.ts` (NULLABLE site_id, plus the audit log which is append-only — reuse the 0005 disjunctive pattern).
- **M1.8e**: `shared/schema.ts` + `shared/admin/schema.ts` remaining tables.

Each sub-milestone is a completable unit with tests. The user can pause between any of them.
