## What was accomplished in v7

### Phase D readiness audit (2026-04-15 evening, STATUS v7)

Three items, all captured in full in `STATUS_2026-04-15_v7.md` § "Phase D readiness audit (the new work in v7)":

**1. Full bypass grep of `server/`.** Every raw `db.*` call against `siteFeatureOverrides` or `featureRolloutHistory` was catalogued. The audit found 100% of the bypasses clustered in a single file: `server/services/feature-flags/pg-storage.ts`. Nothing else in `server/` reads or writes those tables directly.

**2. Storage-layer conversion** (`server/services/feature-flags/pg-storage.ts` + `server/lib/tenant-scoped.ts`):

- **7 per-site call sites** converted to `scoped()` or `withDbSiteScope()`. The evaluator hot path (`evaluator.ts:454` → `siteOverrideStorage.get()`) is now scoped. Under `FORCE ROW LEVEL SECURITY`, it would previously have silently returned zero rows and falsely fallen through to flag defaults — the single most dangerous Phase D break.
- **5 intentional cross-tenant paths** explicitly commented with `INTENTIONAL CROSS-TENANT` + the note that under FORCE they must run as a Postgres superuser role. The app's production DB role must not reach these methods. These are flagged for the user to decide whether to migrate them through a separate admin connection before applying 0004.
- **`withDbSiteScope` is now exported** from `server/lib/tenant-scoped.ts` (was previously a private helper inside the `scoped()` factory). Added docs on when to prefer `scoped()` vs. the escape hatch. No behavior change for existing callers.

**3. Migration 0005 staged** (`migrations/0005_relax_rollout_history_insert_policy.sql`):

Resolves a Phase B policy/design mismatch discovered during the audit. `feature_rollout_history` had an INSERT WITH CHECK requiring strict site_id equality, but the SELECT policy allowed `site_id IS NULL`. Admin-side callers (`FeatureFlagService.createFlag` at `index.ts:212`, `addDependency` at `692`, `removeDependency` at `729`) write global audit events with `siteId === null`, which the Phase B INSERT policy silently rejects whenever the calling role is not a Postgres superuser. Under 0004 (FORCE), the rejection happens to superusers too, breaking the admin UI.

0005 relaxes the INSERT WITH CHECK to match the SELECT:

```sql
WITH CHECK (
  site_id IS NULL
  OR site_id = current_setting('app.current_site_id', true)
)
```

Two alternatives (superuser connection, sentinel siteId) were considered and rejected in the migration file with full rationale. **Read `migrations/0005_relax_rollout_history_insert_policy.sql` before applying** — the user needs to agree with the reasoning before the SQL runs anywhere.

**4. RLS test updated in lockstep** (`tests/rls/rls-isolation.test.ts`):

- Inlined DDL updated to the 0005 form (so the test validates the expected deployed end-state, not the Phase B intermediate state).
- Assertion 10 changed from "INSERT with NULL site_id is rejected" to "INSERT with NULL site_id is allowed (global audit write)".
- New assertion added: "INSERT with mismatched non-null site_id is still rejected" — proves the relaxation only affects the NULL case.
- 12 → 13 assertions total.

**Proceed autonomously:**
- Running the verification commands.
- Writing new RBAC policies, marker migrations, or test assertions that extend existing patterns.
- Fixing type errors you accidentally introduce.
- Updating `MIGRATED_FILES` or `tenant-scoped-registry.ts` as part of a larger migration you're shipping.
- **Converting any newly discovered `db.*` → `siteFeatureOverrides` / `featureRolloutHistory` bypass to `scoped()` or `withDbSiteScope()`** (NEW — this was the v7 work, and the same pattern is now established policy for future encounters).

**Ask first:**
- Anything that touches `package.json` dependencies.
- **Applying 0005 or 0004** (or any future one-way migration).
- Adding a new external service integration.
- Committing or pushing.
- Deleting any file that isn't generated/temporary.
- Any change that would lower the security test count or introduce a typecheck error.
- Re-adding `@anthropic-ai/sdk` or any Anthropic-backed code path.
- **Registering a new multi-tenant table** in `tenant-scoped-registry.ts` (because it implies extending migration 0003 and the RLS test suite — user should know which table and why).
