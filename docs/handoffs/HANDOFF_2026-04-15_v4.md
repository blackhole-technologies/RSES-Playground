### The validateKey cross-tenant problem

The single biggest finding from v9. The SDK auth flow (`apiKeyAuth` middleware → `ApiKeyService.validateKey(key)`) arrives at the server with an opaque key and no site context. The owning site is only learned AFTER the hash lookup. Under migration 0006's strict equality SELECT policy, an unbound `app.current_site_id` session var makes the lookup return zero rows — every valid API key comes back as 401 Unauthorized. The SDK auth flow breaks.

v9 deliberately **does not resolve this in code** because the resolution is a deployment-architecture choice. The three options are all valid and are documented in full at three locations:

1. `migrations/0006_add_api_keys_rls_policies.sql` — full rationale with code sketches.
2. `server/services/api-keys/api-key-service.ts:validateKey` — operational JSDoc pointing at the migration.
3. `tests/rls/rls-isolation.test.ts` — new assertion 3 in the api_keys block explicitly documents the runtime break signal.

**Options:**

- **(a) Superuser connection for validateKey** — add a second DB client wired to a dedicated env var, route validateKey through it. Simplest. Broadens superuser blast radius.
- **(b) Dedicated `app.api_key_lookup` session variable** (recommended) — ship a future migration 0007 that adds a 5th policy: `USING (current_setting('app.api_key_lookup', true) = 'true' OR site_id = current_setting('app.current_site_id', true))`. Wrap validateKey's SELECT in a small helper that sets the flag. Cleaner than (a); survives a future FORCE flip on api_keys.
- **(c) Encode site in the key itself** — changes wire protocol; rejected during M1.8a.

The next session should decide which option to pursue before applying 0006 to production. Staging apply is safe without a decision (staging typically uses a superuser role, which bypasses ENABLE policies — the break only surfaces under a non-superuser role).

### Top priority 2: validateKey deployment decision

Independent of the apply chain. Needs to be resolved before 0006 lands in production under a non-superuser role. My recommendation is option (b) — dedicated session variable — because it keeps validateKey on the main DB client and survives a future FORCE flip on api_keys. Sketch for option (b):

- Migration 0007 adds a 5th SELECT policy on api_keys with the disjunctive form.
- New helper `withApiKeyLookupScope(fn)` in `server/lib/tenant-scoped.ts` that opens a transaction and runs `SELECT set_config('app.api_key_lookup', 'true', true)` before `fn(tx)`.
- `api-key-service.validateKey` switches from raw `db.select()` to `withApiKeyLookupScope(tx => tx.select()...)`.
- New RLS test assertion: "api_keys SELECT with app.api_key_lookup=true returns rows regardless of site".
- The dev-query guard needs an exception for this path — either wrap the `db.select` in a way the guard recognizes, or add a narrow allow-list. Probably the cleanest approach is to have `withApiKeyLookupScope` use the `tx` (which the guard does not wrap), same pattern as `withDbSiteScope`.
