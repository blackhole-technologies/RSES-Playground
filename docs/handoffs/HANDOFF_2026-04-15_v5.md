## What v10 shipped

### Decision: option (b)

v9 surfaced three options for the validateKey cross-tenant lookup under migration 0006's strict equality policy:

- (a) Superuser connection
- (b) Dedicated `app.api_key_lookup` session variable + 5th SELECT policy
- (c) Encode site in key format

v10 chose **(b)** because it is contained (private helper in one file), auditable (grep for `app.api_key_lookup` returns exactly 2 matches — helper + policy), SELECT-only (the bypass does not affect INSERT/UPDATE/DELETE), and survives a future FORCE flip on api_keys.

### Code changes

**`server/services/api-keys/api-key-service.ts`:**

- Added `sql` import from drizzle-orm.
- Added a private `withApiKeyLookupScope<T>(fn: (tx: typeof db) => Promise<T>)` helper above `ApiKeyService`. Opens a Drizzle transaction, runs `set_config('app.api_key_lookup', 'true', true)`, then calls `fn(tx)`. Private by design — the bypass has exactly one legitimate caller, and keeping the helper colocated makes code review trivial.
- Rewrote `validateKey` to wrap the lookup in `withApiKeyLookupScope(async tx => tx.select()...)`. The query body is otherwise unchanged. Since the lookup now runs against `tx` (not the Proxy-wrapped `db`), the M1.3 dev-query guard is transparent to this path — no more dev-mode throws on the SDK auth flow.
- Replaced the multi-paragraph `INTENTIONAL CROSS-TENANT` JSDoc on `validateKey` with a shorter "Cross-tenant lookup model (resolved in migration 0007)" note.

### Migration

**`migrations/0007_add_api_keys_lookup_bypass_policy.sql`** — staged, not applied:

- Single additive `CREATE POLICY api_keys_auth_lookup ON api_keys FOR SELECT USING (current_setting('app.api_key_lookup', true) = 'true')`.
- PERMISSIVE by default; OR-combines with the existing strict policy from 0006.
- Multi-paragraph banner with visibility matrix, security properties, apply procedure, rollback.
- Depends on 0006 being applied first. Reversible via `DROP POLICY IF EXISTS api_keys_auth_lookup ON api_keys`.

### Test coverage

**`tests/rls/rls-isolation.test.ts` — 20 → 23 assertions:**

- `beforeAll` DDL updated to create the 0007 bypass policy alongside the 0006 strict policies.
- The previous "validateKey break signal" assertion rewritten as "fail-closed baseline" (both policies evaluate false when neither session var is set; must return zero rows).
- **New assertion**: "SELECT with `app.api_key_lookup='true'` returns rows regardless of site binding" — the positive bypass behavior.
- **New assertion**: "INSERT with bypass flag AND site binding but wrong site_id rejected" — proves the bypass is SELECT-only.
- **New assertion**: "INSERT with only bypass flag (no site binding) rejected" — fail-closed writes.

### Clarification for v10

- **The bypass helper `withApiKeyLookupScope` is private by construction.** Do NOT move it to `tenant-scoped.ts`. The security model depends on the grep-audit property: the entire codebase must contain exactly one write site for `app.api_key_lookup` and exactly one policy reading it. If a future PR wants to use a similar bypass pattern for a different table, it should duplicate the helper colocated with its own single caller, not share a common export.

- **Do NOT add new writers of `app.api_key_lookup`.** Any such write site must come with a new SELECT policy that is explicitly designed to accept the flag and a grep-audit story that justifies the expansion.
