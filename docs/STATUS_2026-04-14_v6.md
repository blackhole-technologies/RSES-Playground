## What changed since v5

### Phase A — Transaction-scoped site variable in `tenant-scoped.ts`

The existing `scoped(siteId)` helper previously ran queries directly
through the shared `db` client. v6 adds a `withDbSiteScope(siteId, fn)`
helper that opens a Postgres transaction, runs
`SELECT set_config('app.current_site_id', $1, true)` as the first
statement, then executes the caller's function. Every `select` /
`selectOne` / `insert` / `update` / `deleteWhere` method on the
`ScopedQueries` interface now routes through this wrapper.

**Why set_config over SET LOCAL.** `SET LOCAL` is a plain SQL
statement that does not accept bind parameters, which would force
string interpolation of the siteId and create a SQL injection
surface. `set_config('app.current_site_id', $1, true)` is a function
call that accepts parameters. The third argument (`is_local = true`)
scopes the setting to the transaction and auto-resets on COMMIT.
This makes the mechanism safe under the pool's connection reuse
semantics: when the transaction ends, the setting is gone.

**Why a transaction per query.** The alternative — setting the
variable once on pool check-out — is unsafe because the pool reuses
connections across requests. A connection set for site-a and then
returned to the pool would serve a subsequent site-b request with
site-a's binding still active. Per-query transactions are the only
mechanism that guarantees strictly-bounded scope. The cost is
BEGIN/COMMIT overhead per query (sub-millisecond locally; well
under the RLS query-plan overhead).

**Why this is safe without Phase B.** Until migration 0003 is
applied, the `set_config` call sets a session setting that nothing
reads. It is a harmless no-op. Phase A can ship independently of
Phase B. The converse is NOT true — if 0003 is applied to a DB
whose app still runs pre-Phase-A code, the app's queries will
return zero rows (correct but unhelpful). Hence: Phase A first,
then 0003.

### Phase B — `migrations/0003_add_tenant_rls_policies.sql`

Eight policies total. Four on `site_feature_overrides`:

```sql
CREATE POLICY site_isolation_select ON site_feature_overrides
  FOR SELECT
  USING (site_id = current_setting('app.current_site_id', true));
```

Same shape for INSERT (WITH CHECK), UPDATE (USING + WITH CHECK),
and DELETE (USING). These table columns are `NOT NULL`, so the
strict equality policy is sufficient.

Four on `feature_rollout_history`, which has a **nullable** `site_id`
because some rollout events are system-wide (not scoped to a site).
The SELECT policy allows reads on NULL-site rows because global
events are public history. The INSERT WITH CHECK requires a
non-NULL site_id that matches the bound session, so tenants cannot
forge global records to hide cross-tenant activity:

```sql
CREATE POLICY site_isolation_select ON feature_rollout_history
  FOR SELECT
  USING (
    site_id IS NULL
    OR site_id = current_setting('app.current_site_id', true)
  );

CREATE POLICY site_isolation_insert ON feature_rollout_history
  FOR INSERT
  WITH CHECK (site_id = current_setting('app.current_site_id', true));
```

**Why ENABLE and not FORCE.** `ENABLE ROW LEVEL SECURITY` makes
policies apply to non-superuser roles. The app's production DB user
is (should be) a non-superuser, so policies apply there. Superusers
(dev / test / admin tools) bypass the policies entirely. That is
intentional: it means 0003 can be applied to any environment
without breaking existing background jobs, ad-hoc queries, or
admin reconciliation scripts. The FORCE flip — the one-way door —
is a separate migration.

**Rollback.** Pure policy addition. No data changes. Rolling back
is `DROP POLICY … ; ALTER TABLE … DISABLE ROW LEVEL SECURITY` for
each table. Documented verbatim in the migration file header.

### Phase C — `tests/rls/rls-isolation.test.ts`

First DB-touching test in the repository. Design choices:

- **Self-contained.** Does not import `server/db.ts` or go through
  the application's Drizzle layer. Uses `pg.Client` directly. This
  keeps the test independent of the app's `DATABASE_URL` requirement
  and makes it strictly a test of Postgres policy behavior.
- **Isolated schema.** Each run creates a new `test_rls_<uuid>`
  schema via `randomUUID()` (not `Math.random()` — hard rule 4),
  creates minimal copies of the two tables, applies the policy DDL
  inline (copied from 0003), and drops the schema in `afterAll`.
  This means the test cannot collide with real data and parallel
  runs cannot race on each other.
- **Applies FORCE inside the test.** Typical local/CI Postgres runs
  as a superuser, so `ENABLE` alone would be bypassed and the
  policies would be invisible to the test. Applying FORCE inside
  the test validates the end-state (Phase B + Phase D together)
  rather than just the ENABLE state that 0003 ships.
- **12 assertions covering:**
  1. site-a sees only site-a SELECTs
  2. site-b sees only site-b SELECTs
  3. Missing session var returns zero rows (fail-closed)
  4. INSERT for mismatched site is rejected by the policy
  5. INSERT for bound site succeeds
  6. UPDATE for mismatched site affects zero rows
  7. DELETE for mismatched site affects zero rows
  8. site-a sees site-a + global NULL rows in history
  9. site-b does not see site-a rows but does see global rows
  10. INSERT with NULL site_id is rejected by the policy
  11. INSERT for bound site in history succeeds
  12. Empty-string session var returns zero rows

- **`describe.skipIf(!CONNECTION_URL)`** — the suite skips the
  entire block when neither `TEST_DATABASE_URL` nor `DATABASE_URL`
  is set. Vitest reports this as "skipped" in the output, which is
  distinct from pass/fail. The fast `test:security` suite is
  unaffected (it uses a different directory).
- **CI integration is automatic.** `build-and-test` in
  `.github/workflows/ci.yml` already spins up a Postgres service
  container and exposes `DATABASE_URL` to the test step. When
  `npm test` runs, it discovers `tests/rls/rls-isolation.test.ts`
  and runs the 12 assertions against the real Postgres service.
  No CI changes were required for this milestone.

### Phase D — `migrations/0004_force_tenant_rls_policies.sql` (STAGED)

Written to disk but explicitly **not applied**. The file starts
with a multi-paragraph banner:

> ⚠ DO NOT APPLY THIS MIGRATION WITHOUT EXPLICIT USER APPROVAL ⚠

It documents a manual apply procedure:

1. Run `TEST_DATABASE_URL=postgres://… npm run test:rls` against
   staging. Verify all 12 RLS assertions pass.
2. Take a DB snapshot.
3. Apply inside a transaction:
   ```
   BEGIN;
   \i migrations/0004_force_tenant_rls_policies.sql
   -- Run smoke queries against scoped() routes
   COMMIT;
   ```
4. Record the apply in `docs/STATUS-LATEST.md` with environment.

The file itself is two lines:

```sql
ALTER TABLE site_feature_overrides FORCE ROW LEVEL SECURITY;
ALTER TABLE feature_rollout_history FORCE ROW LEVEL SECURITY;
```

Plus rollback notes (`NO FORCE ROW LEVEL SECURITY`).

**Why this is staged not applied.** FORCE is a one-way door in the
sense that every code path bypassing `withDbSiteScope` starts
failing the moment it lands. Every `db.select().from(...)` call on
these tables that never went through `scoped()` will begin returning
zero rows; every bulk admin tool that iterates across sites will
need explicit `SET LOCAL ROLE some_superuser`. Applying without
staging validation risks breaking production. The two-file split
lets the user sequence the rollout at their pace.
