## What shipped between v10 and v11

### Commit `9569f10` — "Harden multi-tenant isolation" (catch-up + M1.8b)

Committed the entire v6–v10 working-tree — M1.4 Phase D readiness, M1.3 dev-query guard, M1.8a api_keys registration, M1.8a-follow validateKey option (b) — as a single squash. On top of that catch-up, M1.8b is new in this commit:

- **Registry:** `socialAccounts`, `socialPosts`, `socialCampaigns` added. Registry count 3 → 6. All three have NOT-NULL `site_id`.
- **Migration 0008** (staged): 12 strict-equality policies — 4 per table (SELECT / INSERT / UPDATE / DELETE) — with the same `site_id = current_setting('app.current_site_id', true)` shape as 0006.
- **Service layer:** 12 per-site methods in `social-media/pg-storage.ts` converted to `scoped()` / `withDbSiteScope()`. 10 by-id methods (`getById`, `update`, `delete`, etc.) were flagged `FIXME` and deferred to M1.8b-follow.
- **Tests:** +9 RLS assertions (23 → 32) — cross-site isolation and policy-shape checks per table.

### Commit `89ca014` — M1.8b-follow (social-media by-id threading)

Resolves the 10 FIXMEs from `9569f10` (plus 5 adjacent call sites) by making `siteId` an optional trailing parameter on by-id storage methods:

- `getById`, `update`, `delete`, `updateStatus`, `updatePlatformStatus`, `getByCampaign`, `updateMetrics` all now accept optional `siteId`.
- When provided, scopes via `withDbSiteScope` / `scoped()` — dev-query guard safe under RLS.
- When omitted (scheduler, publishing queue, analytics collector — all background jobs), falls through to raw `db`. The dev-query guard throws in `NODE_ENV=development`, which is the correct signal for background paths that need explicit superuser or different access patterns.
- Service method signatures updated: `disconnectAccount(siteId, accountId)`, `updatePost(siteId, postId, ...)`, etc.
- **Noteworthy architectural simplification:** the site-scoped wrapper's redundant "list all → find → check ownership → delegate" pattern is replaced with direct `this.siteId` passing. Application-layer ownership checks are removed because RLS at Layer 3 now handles isolation. This is the first concrete case in the codebase of Layer-3 defense-in-depth *replacing* (not merely supplementing) an application-layer check.

### Commit `c2cc119` — M1.8d (rbac registration, NULLABLE `site_id`)

Three rbac-schema tables with NULLABLE `site_id` registered. First use of the disjunctive policy shape from 0005 precedent:

- **Registry:** `userRoles`, `userPermissions`, `auditLogs` added. Registry count 9 → 12.
- **Migration 0010** (staged): disjunctive policies — `WITH CHECK (site_id IS NULL OR site_id = session_var)` — meaning:
  - Tenants see global rows (`site_id IS NULL`) + their own-site rows on SELECT.
  - Tenants can INSERT global rows (this is how admin grants work today — admin mint a global permission, tenants inherit).
  - Tenants **cannot** UPDATE or DELETE global rows (keeps admin grants out of reach of tenant-scoped mutation).
- **`audit_logs` omits DELETE policy** — append-only by design. Deletion requires either superuser connection or a future explicit retention-policy migration.
- **Tests:** +7 RLS assertions (38 → 45) — NULL-visibility, global-insert allowed, mismatched-site rejected, global-update rejected, global-delete rejected, own-site allowed.
- **Service-layer siteId threading** deferred to M1.8d-follow (which later landed as commit `4398ad2`, below).

### Commit `21eab6f` — M1.7-seed (RBAC permission seed closure)

During the M1.7 route marker migration (v5), four resource types were forward-referenced in route markers before their permission keys existed in `initializeDefaults()`. This commit seeds them:

- **14 permission entries** added: `incidents:read/create/update/delete`, `watcher:read/manage`, `automation:read/create/update/delete`, `projects:read/create/update/delete`.
- **Behavior before this commit:** `rbacService.hasPermission` failed closed on unknown keys → non-admins got 403, admins bypassed via `isAdmin`. Delegated RBAC was *silently broken* for these four resource types — nothing threw, nothing logged, and the route markers lint couldn't detect it.
- **Behavior after:** a user granted `incidents:read` can access incident routes without being a full admin.
- **Role propagation:** `super_admin` / `admin` get all (automatic via `allPerms` loop). `editor` gains read on the four new resources. `viewer` gets all `:read` permissions (automatic via `action === "read"`).
- `ResourceType` union in `shared/rbac-schema.ts` extended with `incidents`, `watcher`, `automation`.

## Apply sequencing (updated)

The recommended apply order expands from v10's 4-migration chain to a 7-migration chain. 0003 remains applied; all others are staged:

```
1. 0005 — relax feature_rollout_history INSERT (allow site_id IS NULL)
2. 0006 — api_keys strict equality (4 policies)
3. 0007 — api_keys auth-lookup bypass (1 additional SELECT policy)
4. 0008 — social_media strict equality (12 policies)
5. 0009 — multisite strict equality (12 policies)
6. 0010 — rbac disjunctive (variable-per-table, NULL-aware)
7. 0004 — FORCE on feature_rollout_history + site_feature_overrides
```

Rationale:

- **0005 before 0004** — unchanged from v10. 0004 without 0005 would break admin-side global audit writes.
- **0006 before 0007** — unchanged from v10. 0007's additive SELECT policy assumes 0006's strict-equality policies exist on the same table.
- **0008, 0009, 0010 are independent of 0004 and of each other.** They create policies on *new* tables (social-media / multisite / rbac) — no conflict with the 0004 target (feature-flag tables). Numeric order works.
- **0004 last** — the general principle from v10 holds: apply additive policies first, FORCE flips last. A future migration 0011 would likely FORCE RLS on api_keys (depends on 0006+0007), and a 0012 would FORCE on social_media / multisite / rbac tables (depends on 0008/0009/0010). v11 does not draft either.
- **Note on numbering change from v10:** v10 reserved "0008" for a future FORCE on api_keys. That number is now claimed by M1.8b's social-media policies. The future api_keys FORCE migration will be 0011 or later.
