## The decision: option (b) chosen

Among the three options documented in v9:

| Option | Description | v10 outcome |
|---|---|---|
| (a) Superuser connection | New DB client, new env var, new pool, route validateKey through it | **Rejected** — the new DB client widens the superuser blast radius and doubles the test surface. It also would NOT survive a future FORCE flip on api_keys without a NO FORCE exception specifically for that role. |
| (b) Dedicated `app.api_key_lookup` session variable | Add a 5th PERMISSIVE SELECT policy to api_keys that grants access when the flag is set. Wrap validateKey's query in a helper that sets the flag inside a transaction. | **Chosen** — contained, auditable, survives FORCE, colocated with its one caller. |
| (c) Encode site in the key itself | Changes the wire protocol, requires data migration | **Rejected** — out of scope, and the ff_{tier}_{random} format is public SDK API. |

Reasons option (b) won:

1. **Contained surface area.** The bypass helper is a private function inside `api-key-service.ts`. The bypass policy is a single additional `CREATE POLICY` in migration 0007. A grep for `app.api_key_lookup` across the codebase returns exactly two matches — the helper and the policy. No third match is ever legitimate.
2. **Auditable security model.** The bypass flag is set via a hardcoded literal `'true'` (no request-input path to the value) inside a transaction that has `is_local=true` on the set_config call (auto-clears on COMMIT/ROLLBACK). Cross-transaction leakage is structurally impossible.
3. **SELECT-only.** The bypass grants cross-tenant read. INSERT/UPDATE/DELETE policies from 0006 are unchanged and still require strict `site_id = current_setting('app.current_site_id', true)`. A caller that sets the bypass flag and tries to INSERT a row still fails closed because the bypass scope deliberately does NOT set `app.current_site_id`. Three new RLS assertions validate this invariant.
4. **Survives FORCE.** Under a future migration 0008 that flips FORCE on api_keys, the bypass policy continues to work because it is policy-level, not role-level. Option (a) would need a separate `NO FORCE` exception for the lookup role.
5. **No new infrastructure.** No new DB client, no new env var, no new pool. The helper uses the existing `db.transaction(...)` mechanism that `withDbSiteScope` already relies on.

## What shipped in v10

### Code

**`server/services/api-keys/api-key-service.ts`**:

1. New private helper at the top of the file (above `ApiKeyService`):

   ```ts
   async function withApiKeyLookupScope<T>(
     fn: (tx: typeof db) => Promise<T>
   ): Promise<T>
   ```

   Opens a Drizzle transaction, runs `SELECT set_config('app.api_key_lookup', 'true', true)`, then calls `fn(tx)`. Private to the file — not exported from `tenant-scoped.ts`. Full JSDoc documents the security model including why it is colocated here rather than in the shared helper module.

2. `validateKey` rewritten to use the helper:

   ```ts
   const rows = await withCircuitBreaker(async () =>
     withApiKeyLookupScope(async (tx) =>
       tx.select().from(apiKeys).where(...).limit(1)
     )
   );
   ```

   The query body is unchanged — same WHERE clause, same limit(1), same `updateLastUsed(row.id, row.siteId)` follow-up. Only the client it runs against changes from `db` (Proxy-wrapped) to `tx` (not wrapped). This means:
   - Under migration 0007's bypass policy, production non-superuser roles can execute the lookup.
   - The dev-query guard (M1.3) is transparent to this path because the guard only wraps the top-level `db` export — a tx yielded by `db.transaction()` is never Proxy-wrapped.

3. The old `INTENTIONAL CROSS-TENANT` multi-paragraph JSDoc on `validateKey` was replaced with a shorter "Cross-tenant lookup model (resolved in migration 0007)" note that references the security invariants instead of enumerating options.

4. `import { sql } from "drizzle-orm"` added (needed by the helper's `set_config` call).

**`server/lib/tenant-scoped.ts`**: unchanged. Intentionally NOT a location for the bypass helper — see the code comment in `api-key-service.ts` and the migration 0007 rationale.

Same behavior (both policies evaluate false when neither session var is set), but the comment now frames it as the permanent fail-closed invariant rather than a documentation-of-break.

Sets only the bypass flag, expects all 3 seed rows visible across both site-a and site-b. This is the positive assertion that validateKey will work under the new policy.

Sets BOTH `app.api_key_lookup='true'` AND `app.current_site_id='site-a'`, then tries to insert a row claiming `site_id='site-b'`. Must be rejected by the unchanged INSERT WITH CHECK from 0006. Covers the "buggy caller forgets the site binding" case and proves the fail-closed posture on writes.

## Apply sequencing (updated)

The recommended order changes slightly in v10 — 0007 now sits between 0006 and 0004:

```
1. 0005 — relax feature_rollout_history INSERT
2. 0006 — api_keys strict equality (4 policies)
3. 0007 — api_keys auth-lookup bypass (1 additional SELECT policy)
4. 0004 — FORCE on feature_rollout_history + site_feature_overrides
```

Rationale:
- 0005 before 0004 — unchanged. 0004 without 0005 breaks admin-side global audit writes.
- 0006 before 0007 — 0007's additive policy assumes the 0006 strict policies exist. 0006 alone would break validateKey under non-superuser; 0007 restores the lookup path.
- 0004 last — the FORCE flip on the feature-flag tables is independent of api_keys.

A future 0008 (FORCE on api_keys) can ship whenever. It depends on both 0006 and 0007 being applied — the bypass policy in 0007 survives FORCE the same way the strict policies do. v10 does NOT draft 0008; it can wait until the user explicitly wants FORCE on api_keys in production.

## Security proof of option (b)

Three invariants make option (b) safe. All three are covered by tests.

**Invariant 1: The bypass flag only affects SELECT.**

PostgreSQL applies multiple PERMISSIVE policies with OR semantics for the same command. 0007 adds a second SELECT policy only. INSERT/UPDATE/DELETE policies from 0006 are unchanged and still require strict `site_id = current_setting('app.current_site_id', true)`. Asserted by the two new INSERT-rejection assertions.

**Invariant 2: The bypass flag value is hardcoded and cannot be influenced by request input.**

The helper `withApiKeyLookupScope` contains a literal `sql\`SELECT set_config('app.api_key_lookup', 'true', true)\`` call. The string `'true'` is a compile-time constant. No parameter, no template interpolation, no function argument. Grep confirms: one write site, one policy, no third match.

**Invariant 3: The bypass flag is transaction-local and auto-clears.**

`set_config(..., true)` with `is_local = true` scopes the setting to the enclosing transaction. `db.transaction(async tx => ...)` opens and commits/rolls back a transaction, so the flag never persists beyond a single validateKey call. Cross-transaction leakage under pool connection reuse is structurally impossible.
