## M1.3 тАФ what shipped

### The enforcement mechanism

`server/lib/dev-query-guard.ts` exports `wrapDbWithDevGuard(rawDb, { enabled })`. When `enabled` is true it returns a `Proxy` over the raw Drizzle client with:

- `db.select()` тЖТ wraps the returned builder; intercepts `.from(table)`; throws if the table is in the multi-tenant registry.
- `db.insert(table)` тЖТ throws if `table` is registered.
- `db.update(table)` тЖТ throws if `table` is registered.
- `db.delete(table)` тЖТ throws if `table` is registered.
- Every other property (`transaction`, `execute`, `query`, `$with`, `schemaName`, etc.) passes through via `Reflect.get` unchanged.

When `enabled` is false, the function returns the raw db reference unchanged тАФ zero Proxy overhead. This is the hot path for production and test.

### Why dev-only

The production defense is Postgres RLS (M1.4 Phase B + the pending Phase D). A forgotten `scoped()` call returns zero rows under FORCE, which is loud enough in staging/production. But staging feedback is slow; a developer running locally (with no RLS applied because migrations haven't landed on their local DB) would see no signal at all. The guard gives the local dev loop the same fail-fast behavior that FORCE RLS gives staging.

The guard is NOT enabled in `NODE_ENV=test` because many fixture-seeding tests use raw `db.insert()` calls on tagged tables. Blocking those would require rewriting every fixture to go through `withDbSiteScope`, which is out of scope for M1.3.

### Escape hatches

The guard distinguishes the top-level `db` (wrapped) from a transaction `tx` passed into `withDbSiteScope(siteId, tx => тАж)` (not wrapped тАФ `tx` is a fresh object yielded by `db.transaction`). So:

- `db.select().from(siteFeatureOverrides)` тАФ **blocked in dev**
- `scoped(siteId).select(siteFeatureOverrides)` тАФ OK (uses `tx` internally)
- `withDbSiteScope(siteId, tx => tx.select().from(siteFeatureOverrides))` тАФ OK
- `db.select().from(users)` (untagged) тАФ OK

An intentional cross-tenant admin path that uses raw `db` on a tagged table (e.g. `PgSiteOverrideStorage.getForFeature` in `pg-storage.ts`, already marked `INTENTIONAL CROSS-TENANT` in v7) will throw in dev тАФ that is a feature. The developer should run such paths under `NODE_ENV=test` or production, or refactor to route through a superuser connection.

### Relationship to the static lint

`tests/security/tenant-scoped-imports.test.ts` is still the PR-time gate тАФ it catches missing *imports* without running any code and is cheaper than running the guard. v8 adds the runtime gate as a complement, not a replacement. The two together cover:

- Lint catches: "file imports a tagged table but not the helper."
- Guard catches: "file imports the helper, then also uses raw `db.*` on a tagged table."

Both layers are cheap. Both should stay.

### Refactor bonus

To let the guard and its unit tests import the `isMultiTenantTable` registry without pulling in `server/db.ts` (which throws at module load if `DATABASE_URL` is unset), the WeakMap and three helpers (`registerMultiTenantTable`, `isMultiTenantTable`, `siteIdColumnFor`) were extracted from `tenant-scoped.ts` into a new db-free file `server/lib/tenant-scoped-tables.ts`. `tenant-scoped.ts` re-exports the public functions unchanged, matching the existing `tenant-scope-context.ts` split pattern. No caller outside `tenant-scoped.ts` itself needs to know about the split.

`tests/security/tenant-scoped-imports.test.ts` was updated to add `server/lib/tenant-scoped-tables.ts` to `ALLOWED_UNSCOPED_FILES` тАФ it is part of the helper by construction, same allow-list posture as `tenant-scoped.ts` and `tenant-scoped-registry.ts`.

### Test coverage

New file `tests/security/dev-query-guard.test.ts` тАФ **13 assertions** grouped into four blocks:

1. **Disabled path** (2 tests): returns raw db by identity; lets tagged-table calls through when off.
2. **Enabled path, tagged table** (5 tests): blocks `select().from()`, `insert`, `update`, `delete`; error message contains the fix recipe naming `scoped(siteId)` and `withDbSiteScope`.
3. **Enabled path, untagged table** (4 tests): passes `select().from()`, `insert` through; non-verb properties (`schemaName`, `transaction`) pass through; registry state is unaffected by wrapping.
4. **Proxy integrity** (2 tests): non-`from` builder methods still work; double-wrap does not infinite-recurse.

The tests use a fake `rawDb` with call-logging spies rather than a real Drizzle client, so they run in the DB-free fast suite.

## Bonus finding: the M1.8 gap is much larger than v6 estimated

While building the guard I ran a full grep for `site_id` / `siteId` across `shared/`. The two registered tables (`siteFeatureOverrides`, `featureRolloutHistory`) are a **small minority** of the multi-tenant surface area. Other schema files declaring `site_id` columns:

- `shared/social-media-schema.ts` тАФ 3 tables (social_accounts, social_posts, campaigns) тАФ NOT NULL
- `shared/rbac-schema.ts` тАФ 3 tables (user_roles, user_permissions, audit log) тАФ NULLABLE
- `shared/api-keys-schema.ts` тАФ 1 table тАФ NOT NULL
- `shared/multisite-schema.ts` тАФ 5+ tables (domains, role_assignments, analytics, site_sync with source/target site columns, one other NULLABLE) тАФ NOT NULL on most
- `shared/schema.ts` тАФ 2 tables (one NOT NULL, one NULLABLE)
- `shared/admin/schema.ts` тАФ 2 tables beyond the ones already registered (site_metrics NOT NULL, another NULLABLE)

**That is at least ~15 unregistered multi-tenant tables.** M1.8's scope estimate of "0.5 day" in v6's roadmap is out-of-date тАФ registering all of them, writing policies for each, and adding RLS assertions is more realistically 1тАУ3 days.

v8 deliberately does NOT touch this. The value of shipping M1.3 first is that M1.8's expansion becomes self-verifying under the guard: any new raw-db call on a newly-registered table will fail in dev on the spot instead of waiting for a v7-style manual audit to catch it.
