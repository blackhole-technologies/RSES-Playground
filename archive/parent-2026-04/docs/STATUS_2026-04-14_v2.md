## TL;DR

RSES-Playground is a competent enterprise infrastructure shell with three
strong layers (kernel module system, CQRS/ES core, build/deploy) wrapped
around an aspirational AI/ML feature ring that is largely heuristic-or-mock.
Phase 1 (Foundation) is real and as of v2 has **structural enforcement** for
all three previously-claimed-but-unenforced security properties:

- WebSocket session validation (M0): real session-store lookup, no more cookie-presence no-op.
- Tenant isolation (M0 + M1): query-layer helper, runtime AsyncLocalStorage scope, **CI-gated lint test that fails the build on any file that imports a tagged multi-tenant table without using the helper.**
- Workflow expression DoS (M0): length, depth, and operation-count limits.
- Fail-closed RBAC (M1): handler-marker pattern with admin bypass, audit logging, and "fail closed on rbac service errors" semantics.
- Module init hangs (M1): 30s `Promise.race` budget on initialize/start/stop with `ModuleOperationTimeoutError`. A hung module no longer blocks bootstrap.
- Distributed rate limiting (M1): pluggable backend with in-memory default and Redis-backed implementation that fails open on Redis loss.

## What changed since v1

### M1.1 — Per-module init timeout (kernel registry)

**Files:** `server/kernel/registry.ts`, `tests/security/module-timeout.test.ts`

`initialize()`, `start()`, and `stop()` calls on every module are now
wrapped in `Promise.race` against a 30-second timeout. On timeout the
registry throws `ModuleOperationTimeoutError`, marks the module failed,
and continues with the rest of the dependency graph. Bootstrap can no
longer be blocked by a single hung module.

JavaScript cannot actually cancel the underlying promise — the doc
comment makes this explicit so future maintainers don't assume the hung
module is killed. It's just no longer blocking us.

### M1.5 — Fail-closed RBAC handler markers

**Files:** `server/middleware/rbac-protect.ts`, `tests/security/rbac-fail-closed.test.ts`

Every route handler can now be wrapped in exactly one of:
- `protect(permission, handler)` — auth + specific permission
- `protectAll(perms, handler)` — auth + ALL permissions
- `protectAny(perms, handler)` — auth + ANY permission
- `authRoute(handler)` — auth only, no permission
- `publicRoute(handler)` — no auth, MUST be intentional

Each wrapper stamps a non-enumerable `RBAC_MARKER` symbol on the handler
so a future lint test can statically scan `server/routes/` for unmarked
handlers and fail CI. The deny path:
- Audit-logs every access denial via the existing audit service.
- Fails closed on RBAC service errors (returns 500 rather than allowing).
- Bypasses permission checks for `user.isAdmin` only.

### M1.3 — Tenant scope tracking + structural lint enforcement

**Files:** `server/lib/tenant-scope-context.ts`, `server/lib/tenant-scoped.ts`, `tests/security/tenant-scoped-imports.test.ts`

Three layered changes:

1. **AsyncLocalStorage scope context** in `tenant-scope-context.ts`
   (deliberately split from `tenant-scoped.ts` so it has zero db
   dependency and can be unit-tested without DATABASE_URL).
2. **`scoped(siteId)` runs every query inside `runInTenantScope`** so
   downstream background jobs and event handlers can call
   `assertScoped("caller-name")` to verify the binding propagated.
3. **A lint test scans `server/` for files that import a multi-tenant
   table from `@shared/schema` without also importing
   `tenant-scoped`**. The first run flagged
   `server/services/feature-flags/pg-storage.ts` — a real storage
   backend that uses explicit per-site WHERE clauses and is therefore
   exempt by convention. It's allowlisted with documented reason.

Future tagged tables: add to `MULTI_TENANT_TABLE_NAMES` in the test +
`tenant-scoped-registry.ts` and the lint will enforce immediately.

### M1.2 — Optional Redis-backed rate limiter

**Files:** `server/kernel/rate-limiter.ts` (new), `server/kernel/gateway.ts`, `tests/security/rate-limiter.test.ts`

The gateway's previous in-memory rate limiter silently produced 5x
effective limits on a 5-pod deployment. v2 extracts the limiter behind a
`RateLimiterBackend` interface with two implementations:

- `InMemoryRateLimiter` — fixed window per key, default backend.
- `RedisRateLimiter` — fixed window via INCR + PEXPIRE, distributed-safe.

The factory `createRateLimiter({ redisUrl })` picks Redis when a URL is
provided AND the connection succeeds, otherwise falls back to in-memory
with a warning. Runtime Redis errors fail **open** (allow the request)
rather than fail closed — a rate limiter outage must not take down the
API surface. The gateway's existing in-memory limiter is preserved as
the default; opting in is a one-call upgrade via
`gateway.enableRedisRateLimiter({ redisUrl })`.

The middleware was made async to await the limiter's `check()`. Express
handles async middleware natively as long as rejections are forwarded
to `next(err)`, which we do.
