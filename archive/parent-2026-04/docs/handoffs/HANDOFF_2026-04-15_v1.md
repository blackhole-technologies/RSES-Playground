12. `server/routes/messaging.ts` (33 handlers — the largest file)

**Novel patterns introduced in M1.7:**

- **`publicRoute(withApiPermission(...))` for API-key auth.** `sdk-api.ts`
  doesn't fit the session-based `protect()` model because it uses API
  key authentication, not Passport sessions. The fix: `publicRoute()`
  stamps the marker (metadata only) while a local `withApiPermission`
  composer enforces the per-key permission inside the wrapped handler.
  Behavior-preserving.
- **`validateBody` folded into handlers.** The marker must be the
  *outermost* callable. Chained middleware like
  `router.post(path, validateBody(schema), handler)` doesn't satisfy
  the lint. Fix: inline `schema.safeParse(req.body)` at the top of
  the wrapped handler plus a shared `zodErrorResponse(err)` helper
  that preserves the original 400 shape byte-for-byte.
- **Forward-referenced permission keys for unknown resources.** Four
  files (`incidents`, `automation`, `projects`, `intelligent-watcher-admin`)
  use permission keys that don't yet exist in `shared/rbac-schema.ts`.
  `rbacService.hasPermission` fails closed on unknown keys
  (see `rbac-service.ts:516`), so non-admins get 403 and admins bypass
  via `user.isAdmin`. This preserves the prior admin-only posture and
  remains forward-compatible. The DB seed is tracked as **M1.7-seed**.

### M1.4 Phase A/B/C — Postgres row-level security (2026-04-15, STATUS v6)

Closes the Layer 3 gap in the tenant isolation defense-in-depth model.
Three phases landed in today's session; Phase D is staged but not
applied.

**Phase A — `server/lib/tenant-scoped.ts` transaction wrapping.**
Added `withDbSiteScope(siteId, fn)` which opens a Postgres transaction
and runs `SELECT set_config('app.current_site_id', $1, true)` before
the caller's query executes. Every `ScopedQueries` method
(`select` / `selectOne` / `insert` / `update` / `deleteWhere`) now
routes through it.

- `set_config` over `SET LOCAL` because `SET LOCAL` does not accept
  bind parameters — parameterization is the difference between "safe"
  and "SQL injection surface."
- `is_local = true` (third arg to `set_config`) scopes the setting to
  the transaction and auto-resets on COMMIT/ROLLBACK. Safe under pool
  connection reuse.
- A transaction per query, not a connection-level setting. The
  alternative (set on pool check-out) would stick to the connection
  and bleed across tenants on subsequent checkouts.
- Harmless no-op until Phase B is applied — the setting just sits
  there with nothing reading it.

**Phase B — `migrations/0003_add_tenant_rls_policies.sql`.**
Eight policies total:

- Four on `site_feature_overrides` (NOT NULL site_id → strict
  equality: `site_id = current_setting('app.current_site_id', true)`
  on SELECT / INSERT WITH CHECK / UPDATE USING+WITH CHECK / DELETE).
- Four on `feature_rollout_history` (NULLABLE site_id → SELECT
  `USING (site_id IS NULL OR site_id = ...)` so tenants read global
  events; INSERT `WITH CHECK (site_id = ...)` so tenants cannot forge
  global records; UPDATE/DELETE strict equality).
- **`ENABLE` only, NOT `FORCE`.** Superusers bypass. The app's
  production DB user should be non-superuser, so policies apply there;
  dev/admin tools run as superuser and stay unaffected. Reversible
  via `DROP POLICY` + `DISABLE ROW LEVEL SECURITY`.

**Phase C — `tests/rls/rls-isolation.test.ts`.**
First DB-touching test in the repository. 12 assertions covering:

1. site-a SELECT returns only site-a rows
2. site-b SELECT returns only site-b rows
3. Missing session variable returns zero rows (fail-closed)
4. INSERT for mismatched site is rejected
5. INSERT for bound site succeeds
6. UPDATE for mismatched site affects zero rows
7. DELETE for mismatched site affects zero rows
8. site-a sees site-a + global NULL rows in history
9. site-b does not see site-a rows; does see global rows
10. INSERT with NULL site_id is rejected
11. INSERT for bound site in history succeeds
12. Empty-string session variable returns zero rows

Design:

- Uses `pg.Client` directly, not `server/db.ts` — avoids pulling in
  `DATABASE_URL` at module load.
- Creates a per-run `test_rls_<uuid>` schema via `randomUUID()` (NOT
  `Math.random` — hard rule 4).
- Applies `FORCE RLS` inside the test so the policies actually apply
  under typical superuser test roles. Validates the Phase B + Phase D
  end-state.
- `describe.skipIf(!CONNECTION_URL)` — skips cleanly when neither
  `TEST_DATABASE_URL` nor `DATABASE_URL` is set.
- `afterAll` drops the schema with `CASCADE`. Orphans from crashed
  runs don't collide (UUID uniqueness).

**`npm run test:rls`** added to `package.json`. **No CI workflow
change needed** — the existing `build-and-test` job spins up a
Postgres service container and already exposes `DATABASE_URL` to
`npm test`, which discovers `tests/rls/**` automatically.

**Phase D — `migrations/0004_force_tenant_rls_policies.sql` (STAGED, NOT APPLIED).**
Two-line migration wrapped in a multi-paragraph DO-NOT-APPLY banner
and a manual apply procedure:

1. Run `TEST_DATABASE_URL=postgres://... npm run test:rls` against staging. Verify all 12 assertions pass.
2. Take a DB snapshot.
3. Apply inside a transaction (`BEGIN; \i 0004_...; -- smoke test; COMMIT;`).
4. Record the apply in `docs/STATUS-LATEST.md`.

The migration itself is:
```sql
ALTER TABLE site_feature_overrides FORCE ROW LEVEL SECURITY;
ALTER TABLE feature_rollout_history FORCE ROW LEVEL SECURITY;
```

Rollback: `NO FORCE ROW LEVEL SECURITY` on both tables. This keeps
the Phase B policies active for non-superuser roles.

- Running the verification commands (`npm run check`, `test:security`, `test:rls`).
- Grepping for `scoped()` bypasses and presenting the list.
- Writing new RBAC policies, marker migrations, or test assertions
  that extend the existing patterns.
- Fixing type errors you accidentally introduce.
- Updating `MIGRATED_FILES` or `tenant-scoped-registry.ts` as part of
  a larger migration you're shipping.

**Ask first:**
- Anything that touches `package.json` dependencies.
- Applying `0004_force_tenant_rls_policies.sql` (or any future
  one-way migration).
- Adding a new external service integration.
- Committing or pushing.
- Deleting any file that isn't generated/temporary.
- Any change that would lower the security test count or introduce
  a typecheck error.
- Re-adding `@anthropic-ai/sdk` or any Anthropic-backed code path.
