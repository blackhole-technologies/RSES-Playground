## What changed since v4

### M1.7 — All remaining route files migrated

The fail-closed RBAC marker pattern was introduced in M1.5 with
`admin-audit.ts` as the template. v5 finishes the migration across all
remaining files in the handoff list. For each file, every handler is
wrapped in exactly one of:

- `protect("permission:key", handler)` — auth + specific RBAC permission
- `protectAll([perms], handler)` — auth + ALL listed permissions
- `protectAny([perms], handler)` — auth + ANY listed permission
- `authRoute(handler)` — auth only, no specific permission
- `publicRoute(handler)` — intentional no-auth

### Permission key strategy for unknown resources

Two files referenced permissions that don't yet exist in the
`ResourceType` enum in `shared/rbac-schema.ts`:

- `incidents.ts` → `incidents:{read,create,update}`
- `intelligent-watcher-admin.ts` → `watcher:{read,manage}`
- `automation.ts` → `automation:create` (read routes use `authRoute`
  since their original posture was any-authenticated)
- `projects.ts` → `projects:{read,create,update}` (listed in ResourceType
  but no permission records seeded yet)

The handoff said to use `authRoute()` when no permission exists, but
that would have **regressed** the admin-only posture of the original
`incidents.ts` (which used `requireAdmin`) and the unmounted
`intelligent-watcher-admin.ts` (which had no auth at all — using
`authRoute` would relax the now-correct admin gate). Instead, the
migration uses forward-referenced permission keys: `rbacService`
fails closed on unknown keys, so non-admins are denied, and admins
bypass via `user.isAdmin`. This preserves the prior auth posture
today and becomes strictly stronger once the permissions are seeded.

**TODO in each file** logs the DB seed as follow-up work (M1.7-seed,
tracked in ROADMAP v5).

### Routes that had validateBody middleware chains

Four files used `validateBody(schema)` chained between the path and
the handler:

- `server/routes/assistant.ts` (5 routes)
- `server/routes/messaging.ts` (11 routes)

The marker lint requires the **outermost** callable in the handler
slot to be a recognized marker. `validateBody(...)` as a chained
middleware doesn't satisfy this. Rather than extending the marker
system, the migration folds validation into the wrapped handler via
`schema.safeParse(req.body)` + a shared `zodErrorResponse(err)` helper
that preserves the original 400 response body byte-for-byte:

```ts
{
  error: "Validation Error",
  code: "E_VALIDATION",
  details: err.errors.map(e => ({ field: e.path.join("."), message: e.message })),
}
```

For three routes in `messaging.ts`, the parsed Zod data had stricter
optionality than the service method signatures expected. The fix was
to reassign `req.body = parsed.data` after validation (mirroring what
the original `validateBody` middleware did), so the call-site types
remained `any`. This is behavior-preserving — the original code was
already untyped at the call site.
