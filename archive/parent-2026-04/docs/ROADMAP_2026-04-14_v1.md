# RSES-Playground — Roadmap

**Version:** 1

**Companion doc:** `docs/STATUS_2026-04-14_v1.md` (current state)
**Vision doc:** `docs/plans/CMS-MASTER-PLAN-FINAL.md` (24-month aspiration)

> This roadmap is a **bridge** between today's real implementation (~Phase 1.5) and the Master Plan
> vision (Phases 1–5). It is sequenced by leverage, not by feature glamour. Items are dated relative
> to the start of focused execution, not by calendar.

---

## Sequencing Principles

1. **Honesty before scope.** Every feature ships with a doc update that names what is real and what is
   stubbed. No more dueling status docs.
2. **Enforcement before expansion.** Security architecture is mostly defined; the gaps are in
   enforcement. Close those before adding new surface area.
3. **One real AI path, then a second, then a third.** Wire the SDK once, then reuse it. Don't fan out
   across services until the first integration has shipped, been tested, and has a token budget.
4. **Delete before adding.** File-watcher sprawl, dead .bak files, dueling master plans — they cost
   review attention every session. Cleanup is a feature.
5. **Tests are part of the deliverable.** No "feature done, tests later." Security tests in particular.

## Milestone 1 — Foundation Hardening (2 weeks)

**Goal:** Make Phase 1 actually production-ready for a single-tenant deployment.

| # | Item | Owner | Acceptance |
|---|---|---|---|
| 1.1 | Add per-module init timeout to `server/kernel/registry.ts` | Kernel | Module init that exceeds 30s is killed and reported in health, bootstrap continues |
| 1.2 | Replace in-memory rate limiter in `server/kernel/gateway.ts` with optional Redis backend | Kernel | If `REDIS_URL` is set, limiter uses Redis; otherwise falls back to in-memory with a warning |
| 1.3 | Add a Drizzle pre-query hook (or extend `tenant-scoped.ts`) that *automatically* injects `site_id` for tagged tables | Data | A naive `db.select().from(taggedTable)` fails fast in dev, succeeds in prod with auto-injected scope |
| 1.4 | Add Postgres row-level security policies for top 3 multi-tenant tables (configs, projects, content) | Data + DBA | RLS policies block cross-site reads at the DB layer; tests cover bypass attempts |
| 1.5 | Replace `requireAuth` per-route pattern with an `enforceRBAC(permission)` middleware that fails closed | Security | Routes without explicit RBAC declarations 403 by default, controlled by an allowlist |
| 1.6 | Wire `npm run test:security` into a pre-commit hook and CI gate | DevEx | CI fails if any security test fails |
| 1.7 | Add `npm run test` to CI; surface coverage on PRs | DevEx | Coverage HTML in artifacts, threshold 70% on `server/lib/`, `server/auth/`, `server/middleware/` |
| 1.8 | Document the developer "trust model" in `docs/security/TRUST-MODEL.md` | Security | Single page explaining what the kernel sandbox does and does not protect against |

**Exit criteria:** A new engineer can clone, run `docker compose up`, and have a working dev environment
in under 10 minutes. Security tests are green and gating CI. Tenant isolation is enforced at three
layers (request, query, DB).

## Explicit Non-Goals (For Now)

These are listed so the team doesn't accidentally drift into them:

- ❌ **Quantum-ready features.** Remove from all marketing copy. Revisit when there is a customer ask
  and a credible PQC library to integrate (e.g., `@noble/post-quantum`).
- ❌ **Hosting untrusted third-party plugins.** The kernel sandbox is not designed for this. Re-evaluate
  only after a hard ask + budget for `isolated-vm` or worker_threads + signed-code verification.
- ❌ **Building our own LLM training pipeline.** Use embeddings and prompted Claude. Don't fine-tune.
- ❌ **A new editor.** Monaco is sufficient.
- ❌ **GraphQL gateway.** REST + Zod schemas + OpenAPI generation are enough for the current scale.
- ❌ **Microservices split.** This is a monolith with module boundaries. Keep it that way until there
  is a deployment-driven reason to split.
